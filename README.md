## Experimental setup

Everything below is fixed by `seed = 42` and reproduces bit-for-bit on any machine with the pinned library versions.

### Task & data-generating process

The task is **binary sentiment classification** (positive / negative). The dataset is fully synthetic, constructed from a small vocabulary of slot fillers assembled into sentence templates, so the decision boundary is known by construction. The generator (`generate_data.py`) works as follows:

1. **Vocabulary.** Slot-filler lists are hard-coded: 10 common nouns (`ITEMS`), 10 positive adjectives (`POS_ADJS`), 10 negative adjectives (`NEG_ADJS`), 6 intensifiers (`INTENSIFIERS`). Combined with connector words fixed in the templates, the full vocabulary seen in training is **58 tokens** (confirmed by `results/metrics.json: vocab_size = 58`), plus the two special tokens `<pad>` (id 0) and `<unk>` (id 1).
2. **Templates.** Six template families — three plain (positive/negative without `not`) and two pairs of negated templates — produce sentences of varying surface form but the same underlying rule: *the label is determined by the polarity-bearing adjective AND whether `not` is present*.
3. **Label assignment.** A single `random.Random(seed)` instance draws `use_neg` (Bernoulli with `negation_ratio`), then `label` (Bernoulli 0.5 for positive/negative), and finally selects the matching template family. This ensures labels are balanced across all structural slices.
4. **Dataset size.** `n_train = 1500`, `n_test = 400`. Each set is generated by an independent loop from the same seeded RNG, so the train and test distributions are i.i.d. draws from the same process.

| Parameter | Value | Why it's set this way |
|---|---|---|
| `n_train` | 1500 | Enough examples for a small transformer to converge in a handful of epochs without GPU. |
| `n_test` | 400 | ~20% of the data; enough to get stable per-slice (plain/negated) accuracy estimates. |
| `negation_ratio` | 0.25 | 25% of examples require the model to track `not`; 75% are plain. This mirrors plausible real-world negation rates and lets a bag-of-words model score 75% by ignoring `not` entirely. |
| `seed` | 42 | One seed drives all random draws: RNG used in `generate()`, template selection, slot filling, and label assignment. |

### Tokenization & input representation

- **Whitespace tokenizer.** Tokens are split on whitespace only — no subword, no lowercasing, no punctuation stripping. The sentence `"The movie was really awful ."` becomes `["The", "movie", "was", "really", "awful", "."]`.
- **Vocabulary construction.** Built from the training set only (`vocab_from(train)` in `generate_data.py`). Words not seen in training map to `<unk>` (id 1) at inference time.
- **Sequence length.** `max_len` is computed as `max token count in train ∪ test + 2` (safety margin), which produces `max_len = 14` for the default config (confirmed in `results/metrics.json`).
- **Padding.** Sequences shorter than `max_len` are right-padded with token id 0 (`<pad>`). A key-padding mask is passed to the transformer so pad positions are ignored in attention.

### Full transformer architecture

Built on `nn.TransformerEncoder` for the attention computation, with all other components hand-rolled:

```
Input ids  (B, T)
  │
  ├─ Token embedding     nn.Embedding(58, 64, padding_idx=0)
  ├─ + Positional embedding  nn.Embedding(32, 64)   ← learned, not sinusoidal
  │      [actual max_len used at runtime is 14]
  │
  ├─ TransformerEncoderLayer × 2
  │     ├─ Multi-head self-attention:  d_model=64, n_heads=4
  │     ├─ Feed-forward:               dim_feedforward=256  (64 × 4)
  │     └─ Dropout:                    0.1
  │
  ├─ Mean-pool over non-pad token positions
  └─ Linear head: 64 → 2 logits (negative / positive)
```

| Component | Setting | Note |
|---|---|---|
| Embedding dim `d_model` | 64 | Small but sufficient; keeps the model well under 100K parameters. |
| Attention heads `n_heads` | 4 | Each head attends with 16-dim keys/queries/values. |
| Encoder layers `n_layers` | 2 | One layer can handle local context; the second enables composition across clauses. |
| Feed-forward dim | 256 | Standard 4× expansion of `d_model`. |
| Positional encoding | Learned `nn.Embedding` | Not sinusoidal; learned from scratch. Works because sequences are short and the vocabulary of positions is tiny (≤ 14). |
| Dropout | 0.1 | Applied inside each `TransformerEncoderLayer`. |
| Pooling | Mean over non-pad tokens | No `[CLS]` token; avoids the need to reserve a special position. |
| Output | Linear(64 → 2) | Binary cross-entropy via `F.cross_entropy`. |

### Training hyperparameters

| Hyperparameter | Value | Note |
|---|---|---|
| Optimizer | Adam | Default PyTorch `torch.optim.Adam`. |
| Learning rate | 1e-3 | Fixed; no scheduler, no warmup. |
| Batch size | 32 | Full shuffle each epoch via `DataLoader(shuffle=True)`. |
| Epochs | 8 | Loss collapses by epoch 2; remaining epochs are confirmatory. |
| Loss function | Cross-entropy | `F.cross_entropy` over 2 logits. |
| Gradient clipping | None | Not applied. |

### Train / val / test split

There is no explicit validation split. `generate_data.py` produces two independent sets:

| Split | Count | Role |
|---|---|---|
| Train | 1,500 | Model fitting |
| Test | 400 | Holdout evaluation (plain: ~300, negated: ~100) |

All accuracy figures reported are on the **400-example test set** the model never saw during training.

### Environment

`python ≥ 3.10` · `numpy ≥ 1.24` · `matplotlib ≥ 3.7` · `torch ≥ 2.0`

No HuggingFace or scikit-learn dependency; the transformer is implemented entirely in PyTorch.

### 1. The dataset — examples

![samples](assets/01_samples.png)

Six template families produce the data:

- **Plain positive**: `"The {item} was {intensifier} {pos_adj} ."`
- **Plain negative**: `"The {item} was {intensifier} {neg_adj} ."`
- **Negated → positive label**: `"It was not {neg_adj} at all — the {item} was {pos_adj} ."`
- **Negated → negative label**: `"The {item} was not {intensifier} {pos_adj} , I really disliked it ."`
- (and two more "what a … that was" variants)

The label is determined by the *combination* of polarity-bearing adjective and whether `not` is present. A bag-of-words model would fail the negated examples; a model that pays attention to `not` should solve them.

Loss collapses by epoch 2 and accuracy plateaus at 100% on both splits. The gap between train and test accuracy is essentially zero — no overfitting because the dataset's underlying rule is fully learnable and the model has just enough capacity.

A flawed bag-of-words baseline (not shown here, but trivial to confirm) would score ~75% overall by guessing "positive adjective → positive label" — meaning it would be near-random on negated examples.

### 5. Attention — where the model actually looks

![attention](assets/05_attention.png)

This is the most informative figure: layer-0 attention (averaged over heads) on three negated examples. Reading horizontally — for each row token, the heatmap shows which other tokens it attended to.

- In the leftmost example (positive label, structure: "The trip was not quite disappointing, I actually enjoyed it"), the `not` row attends strongly to `it` and `enjoyed` — consistent with the model learning that `not` reverses the polarity of nearby adjectives.
- In the rightmost (negative-label) example, `at`, `all`, and the dash all attend strongly to `miserable` — the model is forming a representation of the second clause, where the actual polarity lives.
- Notice that in *every* example the model attends across the negation: it doesn't just look at the first adjective.

This is the qualitative evidence that the transformer is genuinely *reading* the sentence, not pattern-matching shallow keyword cues.

## Validation methodology

### Metrics

The task is binary classification, so the primary metric is **accuracy**, reported both overall and per structural slice:

| Metric | Definition | Reads as |
|---|---|---|
| **Accuracy** | $\frac{\text{correct predictions}}{\text{total predictions}}$ | Fraction of test examples classified correctly. A random baseline scores 0.50; a bag-of-words baseline that ignores `not` scores ~0.75 (it gets all plain examples right and random on negated). |
| **Per-slice accuracy** | Same formula, restricted to `struct == "plain"` or `struct == "negated"` | The **negated-slice accuracy is the diagnostic metric**: 100% means the model tracks `not`; ~50% means it ignores it. |
| **Confusion matrix** | $\text{CM}[i, j] = \#\{\text{true}=i, \text{pred}=j\}$ | Shows whether errors are asymmetric (one class systematically mis-classified). |

No F1 or AUC is reported: labels are balanced by construction (`label ~ Bernoulli(0.5)` in the generator), so accuracy and macro-F1 are equivalent here.

### Attention maps — what they do and don't tell you

The `05_attention.png` figure shows layer-0 self-attention weights (averaged over 4 heads) on three negated test examples. A note on interpretation:

**What you can read from attention maps:** which token pairs the model *routes information between* at layer 0. A heatmap where `not` attends strongly to the polarity adjective is consistent with the model encoding negation correctly.

**What you cannot conclude from attention maps:** that the attention pattern *causes* the correct prediction. Attention weights are a byproduct of the dot-product scoring function, not a causal explanation. A model can achieve perfect negation accuracy while showing unintuitive attention patterns, and vice versa. The attention visualization here is a *sanity check*, not a mechanistic proof — it answers "does the routing look plausible?" not "is this the only mechanism the model uses?".

### Train-vs-test gap

From `results/metrics.json`, the learning curve shows:

- Epoch 1: train acc = 80.7%, test acc = 99.0%
- Epoch 2: train acc = 99.6%, test acc = 100.0%
- Epochs 3–8: both train and test stay at 100.0%

The test accuracy surpassing train accuracy at epoch 1 is not unusual: the task is fully learnable, the test examples are drawn from the same distribution, and the stochastic batch order means the model sees all training examples unevenly within the first epoch. By epoch 2 there is no measurable gap.

### Reproducibility

- **Determinism.** `torch.manual_seed(42)` is set at the top of `train_model()`. `generate_data.py` uses `random.Random(42)` for the data. Together they make the run fully reproducible given identical library versions.
- **Seed-sensitivity caveat.** Because the task is *fully learnable* (perfect accuracy is the theoretical ceiling), the reported numbers do not depend on the seed in the usual sense — any seed will converge to 100% test accuracy if the model runs long enough. The epoch at which it converges may vary by one or two epochs with a different seed.

## What's actually happening

### Architecture (about 60K parameters)

```
Input ids (B, T)
  │
  ├─ Token embedding         (vocab → 64)
  ├─ + Positional embedding  (T → 64)
  │
  ├─ Encoder layer 1: multi-head self-attention (4 heads) + feed-forward (256)
  ├─ Encoder layer 2: multi-head self-attention (4 heads) + feed-forward (256)
  │
  ├─ Mean-pool over non-pad tokens
  └─ Linear head → 2 logits
```

Built with `nn.TransformerEncoder` for the attention plumbing, but the embedding lookup, positional encoding, mean-pooling, and classifier head are all hand-rolled — same shape as BERT, just *much* smaller.

### Why a transformer (rather than an LSTM)

Negation is a *long-distance* dependency: the polarity of `terrible` in `"it was not terrible at all — the song was excellent"` depends on the `not` four tokens earlier *and* on the structurally-distant `excellent`. RNNs handle this by passing a hidden state forward; transformers handle it by attending directly across positions in one step. For tasks like this, attention's $O(T^2)$ cost is justified by its ability to wire any-to-any relations in a single layer.

### Why the task is "easy" by design — and why that's good

The signal is *fully learnable* given attention: there exists a 2-layer attention pattern that achieves 100% accuracy. The point of this project is **not** "prove a transformer is the only thing that works" — Naive Bayes would fail, but a proper bag-of-bigrams that captures `not <adj>` would also do well. The point is to **build a transformer end-to-end** (tokenization → embeddings → attention → pooling → loss) in code that fits on one screen, and to verify with attention plots that it learned the right thing.

For a harder showcase, replace `negation_ratio=0.25` with a more adversarial structure (sarcasm, double negation, distant negation) and watch the same architecture struggle. That's where you'd reach for fine-tuning a pretrained DistilBERT — which is the natural next project.

## Notes on methodology & limitations

Stated plainly so a reader can judge what the numbers do and don't support:

- **The vocabulary and task are far simpler than any real NLP problem.** The model sees exactly 58 unique tokens, all from pre-specified lists of 10 items, 10 adjectives per polarity, and 6 intensifiers. There is no morphology, no rare words, no out-of-vocabulary generalization challenge, and no ambiguity in the signal. A real sentiment classifier must handle tens of thousands of word types, sarcasm, domain shift, and genuinely ambiguous cases — none of which appear here.
- **Synthetic structure may be trivially learnable by weaker means.** The templates are deterministic enough that even a bigram model keyed on `"not <adj>"` would score very well. The attention visualization is consistent with the transformer using `not` correctly, but it does not rule out the possibility that the model exploits other template artifacts (e.g., the presence of ` — ` as a reliable negation marker in two of the four negated templates). The clean 100% accuracy on negated examples is a necessary but not sufficient condition for genuine negation understanding.
- **No validation split or hyperparameter search.** The learning rate (1e-3), batch size (32), and epoch count (8) are fixed illustrative values, not the result of any cross-validated tuning. On a real task these choices would need to be justified or searched over a held-out development set.
- **Attention weights are not a causal explanation.** The attention heatmaps in `05_attention.png` show which token pairs the model routes information between at layer 0, averaged over all 4 heads. This is a useful sanity check but attention ≠ attribution: the pattern does not prove that `not`-to-adjective attention is the *mechanism* by which the model classifies correctly. Gradient-based attribution (e.g., integrated gradients) or ablation studies would be required to make that claim.
- **100% accuracy is not surprising given the setup.** A dataset of 1900 total examples drawn from 6 sentence templates with a 58-token vocabulary is not a challenge for a 60K-parameter model. The result validates that the implementation is correct, not that the architecture is powerful. The interesting question for future work is where the same architecture fails — which the `negation_ratio` and adversarial-template knobs are designed to explore.

## What I learned

- **The right metric for negation handling is per-slice accuracy, not overall accuracy.** A model can score 75% overall by ignoring `not` entirely (since 75% of examples here are non-negated). That sounds like a working classifier; in reality it's broken in the most diagnostic 25%.
- **Attention plots are the cheapest interpretability tool.** Three lines of code to extract them, and they give an immediate visual answer to "is the model looking at the right tokens?". For toy tasks like this, the heatmaps are essentially a debugger.
- **A 60K-parameter transformer is enough for surprisingly hard NLP toy tasks.** No pretraining, no positional sinusoids (just learned embeddings), no warmup, no scheduler. The transformer's expressive power per parameter is high — what makes BERT-class models hard is the *scale* of language they have to model, not the architecture.
- **Building a transformer once from scratch makes BERT/RoBERTa/etc. demystified.** Every advanced model in this family is "this same skeleton, but bigger, with pretraining objectives, and with more careful initialization."
